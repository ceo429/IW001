## 1. 프로젝트 개요

### 1.1 배경 및 목적
IOTWORKS는 다수의 IoT 장소(매장·오피스·공장 등)에 스마트 디바이스(스위치·조명·플러그·센서·허브 등)를 설치하고, 운영 후 발생하는 유지보수·AS 인입·기기 모니터링·고객 응대·견적/수주 등 전 과정을 다루는 사업을 수행하고 있다.

기존에는 헤이홈(Hey Home) 등 외부 플랫폼을 이용해 기기를 관리하고, 견적/AS/일정 등은 엑셀과 메신저로 분산 관리하고 있어 다음과 같은 문제가 있었다.

- 장소·기기·고객·프로젝트·견적 정보가 분산되어 한눈에 파악 어려움
- AS 인입 건과 유지보수 이력의 단절
- 견적 작성 시 품목 단가·재고·할인율을 매번 찾아야 하는 비효율
- 권한 분리 없이 모든 직원이 동일 시트를 보는 보안 이슈

본 프로젝트는 이러한 문제를 해결하기 위해 **단일 통합 대시보드**(Single-File HTML 데모 → 향후 SPA/서버 연동)를 구축하여, 운영·유지보수·영업·관리자 전 부문이 동일한 데이터를 기반으로 협업할 수 있는 환경을 제공하는 것을 목표로 한다.

### 1.2 핵심 가치 (KPI)
| 지표 | 목표 |
|---|---|
| 기기 온라인율 가시성 | 실시간 95% 이상 유지 모니터링 |
| AS 평균 응답 시간 | 24시간 이내 |
| 견적 작성 소요 시간 | 기존 30분 → 5분 이내 |
| 유지보수 이력 통합률 | 100%(엑셀 대비) |
| 권한 통제 | 4단계 역할 기반 |

### 1.3 주요 사용자(Persona)
| 역할 | 주요 사용 페이지 | 권한 수준 |
|---|---|---|
| **최고관리자(admin)** | 모든 페이지 | 전체 권한 |
| **매니저(manager)** | 운영·분석·고객·재무 | 관리자 설정/감사로그 제외 |
| **엔지니어(engineer)** | 장소별 현황·공간매핑·프로젝트·AS·유지보수 | 기술/운영 한정 |
| **뷰어(viewer)** | 대시보드·통계·고객·견적·품목 | 읽기 전용 |

## 2. 시스템 구성

- **Frontend**: 단일 HTML 파일(`demo/index.html`, ≈7,645 lines), Vanilla JavaScript, CSS
- **Font**: Pretendard (jsDelivr CDN)

## 4. 핵심 기능 명세

### 4.1 모니터링

#### 4.1.1 대시보드
- 4개 KPI 카드(전체 계정·총 장소·총 기기·온라인율)
- 기간별 도입 추이(연도별)
- 최근 알림 5건
- 즐겨찾기 페이지 바로가기

#### 4.1.2 헤이홈 계정 관리
- 연도(계약기간) 그룹화 카드 뷰
- 토큰 유효/만료 표시
- 카드 클릭 → 해당 계정의 장소 목록(account-homes)

#### 4.1.3 장소별 현황
- 카드 그리드(장소명·기기 수·온라인율·오프라인 강조 테두리)
- 카드 클릭 → 장소 상세(home-detail) → 기기 상세(device-detail)
- 검색·필터(상태/유형) 지원
- 공간매핑 바로가기 버튼

### 4.2 운영

#### 4.2.1 공간매핑 (Spatial)
- 장소별 다층 평면도 SVG 렌더링
- 디바이스 마커 배치/이동/삭제 (편집 모드)
- 룸(영역) 추가/리사이즈/회전
- 줌(0.5×~3×), 오프라인 기기만 보기 필터
- 디바이스 속성판(이름·유형·배터리)

#### 4.2.2 프로젝트 & 태스크
**3가지 뷰 모드**
- **칸반**: 대기/진행/완료 컬럼
- **카드**: 프로젝트별 카드 그리드
- **캘린더 (Google Calendar 스타일)**:
  - 일/주/월/일정 4개 서브 뷰
  - 미니 캘린더 사이드바
  - 시간대 그리드 (주/일 뷰)
  - 현재 시각 라인 표시
  - 클릭 → 해당 날짜 태스크 모달

**태스크 CRUD**
- 제목·우선순위·상태·담당자·기한·연관 장소
- 담당자 다중 선택 필터
- 오늘의 태스크 위젯 (담당자 필터 드롭다운)

#### 4.2.3 AS 인입건
- 인입 상태별 카드(접수/진행/완료)
- 증상·원인·조치 카테고리
- 상세 모달(현장명·증상·조치 내역)

### 4.3 재무 (신규 추가, eCount ERP 벤치마킹)

#### 4.3.1 견적서 (Quotes)
**작성·계산 기능**
- 거래처·프로젝트·장소·담당자·견적일·유효기간 입력
- **품목 라인 추가 방식 3가지**
  1. 카탈로그 검색으로 추가
  2. **빈 줄 추가 후 검색으로 선택** (NEW)
  3. 엑셀(CSV/TSV) 일괄 업로드
- 수량·단가·할인율 입력 → 실시간 합계 계산
- VAT 자동 계산 (기본 10%)

**할인율 일괄 적용**
- 전체 일괄 / 선택 품목 / 고객사별 지정 할인율 (3-mode)
- 고객사별 할인율 영구 저장 (CUSTOMER_DISCOUNT_RATES)

**다중 양식 (4종 템플릿)**
| ID | 명칭 | 특징 |
|---|---|---|
| `tpl-standard` | 표준 견적서 | 기본 양식, 정보표 + 품목표 + 도장 |
| `tpl-simple` | 간이 견적서 | 미니멀, 한 줄 수신자 |
| `tpl-detail` | 상세 견적서 | 규격·설명·납품/결제/계좌 조건 |
| `tpl-tax` | 세금계산서형 | 거래명세서 스타일, 등록번호 강조 |

**부가 기능**
- 직접 속성 편집 (공급자/공급받는자/납품·결제 조건)
- 복제 / 수주 전환(재고 자동 차감 + 프로젝트 태스크 자동 생성)
- PDF 출력 (`window.open` + `window.print`)
- 발송/승인/거절 상태 변경

#### 4.3.2 품목 관리 (Products)
- 15종 IoT 제품 카탈로그
- 카테고리: 스위치/허브/플러그/센서/DC/기타/시공/미디어
- 모델·단위·단가·재고·최소재고·공급사
- KPI: 총 품목·재고가치·재고부족·카테고리 수

- CRUD 모달

**🛡️ 섹션 권한 관리**

**👤 계정 관리** (NEW)

## 6. UI/UX 가이드라인

### 6.1 디자인 원칙
1. **다크 퍼스트**: 모니터링 중심 사용 환경 고려
2. **글래스모피즘 아이콘**: 브랜드 일관성 + 가독성
3. **컴팩트 정보 밀도**: 한 화면에 최대 정보 노출
4. **컬러 코드**: 녹(정상)·앰버(주의)·적(경고)·청(정보)·자(특수)
5. **모바일 대응**: 사이드바 슬라이드, 반응형 그리드

### 6.2 공통 컴포넌트
- **카드(Card)**: KPI / 장소 / 고객사 / 프로젝트
- **태그(Tag)**: 상태/우선순위 표시 (5색)
- **모달(Modal Overlay)**: `z-index 2000`, `.show` 클래스로 페이드 인
- **토스트(Toast)**: 우상단 4종(success/error/warning/info)
- **사이드바(Sidebar)**: 즐겨찾기 + 그룹 메뉴, 드래그 정렬
- **검색바**: 통합 검색(기기/장소/프로젝트/견적/품목)

### 6.3 상호작용 패턴
- 카드 hover → 그림자/테두리 강조
- 행 hover → 배경색 변경
- 모달 외부 클릭 → 닫기
- ESC 키 → 모달 닫기 (구현 예정)
- 키보드 단축키(Cmd+K → 검색, ?  → 도움말) (구현 예정)

### 9.1 로컬 실행
```bash
cd "대시보드/aPI 유지보수대시보드/demo"
npx serve -l 5173
# 브라우저에서 http://localhost:5173 접속
# 로그인: admin@iotworks.kr / 비밀번호 아무거나
```

### 9.2 코드 구조 (단일 HTML)
```
<head>
  <style>          // CSS 변수, 다크/라이트 테마, 컴포넌트 스타일
  </style>
</head>
<body>
  <div class="login-screen">  // 로그인 화면
  <div class="sidebar">       // 사이드바 + 즐겨찾기
  <div class="main">          // 페이지 컨테이너
    <div id="page-dashboard"> // 각 페이지 div
    <div id="page-accounts">
    ...
  </div>
  <script>
    // 1) DEMO DATA       (라인 880~1050)
    // 2) 인증/네비게이션 (라인 1048~1500)
    // 3) 페이지 렌더러   (라인 1527~7295)
    //    - renderDashboard / renderAccounts / renderAdmin / ...
    // 4) 모달/토스트/유틸
  </script>
</body>
```

### 9.3 새 페이지 추가 절차
1. `TABS` 배열에 항목 추가 (`{id, label, icon, group}`)
2. `FIXED_GROUP_ORDER`에 그룹이 없으면 추가
3. `PAGE_TITLES`에 한글 제목 추가
4. `<body>`에 `<div id="page-{id}" class="page">` 컨테이너
5. `function render{Id}()` 작성
6. `navigate()` switch에 case 추가
7. `ALL_PERMISSION_PAGES`에 페이지 추가 (권한 관리 대상)
8. 통합 검색 결과에 포함 (선택)

### 9.4 데이터 모델 변경 시 주의
- DEMO 단계는 모든 데이터가 메모리(`const`/`let`)에 있음
- 페이지 새로고침 시 초기화 → localStorage 영속화 필요한 항목만 저장
- 운영 단계에서는 서버 API로 교체
